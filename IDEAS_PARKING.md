## ­ЪЈЌ№ИЈ ARCHITECTURAL RULE Рђћ All Media Through Cloudinary

**Binding rule for every module from Session 20 forward.**

**The rule:**
> All media (audio, photo, video, PDF, doc, xlsx, csv, drawing) is uploaded to Cloudinary. Metadata is stored in Supabase. The Cloudinary URL lives on a `media_analyses`-style table. Every row has `customer_id` AND `project_id` foreign keys (or explicit nulls justified in the row's notes). Nothing is stored only in someone's email, Dropbox, local filesystem, or third-party SaaS without a corresponding row in the database.

**Why this matters:**
- AI search/analysis works on the database, not on Dropbox
- Customer page (`/customers/[id]`) and future verticals' dashboards can query a unified library view in milliseconds
- Cloudinary handles upload, transformations (thumbnails, audio waveforms, format conversion), CDN delivery, and download URLs natively Рђћ we don't reinvent any of that
- Every artifact is automatically findable, automatically organized by customer/project, automatically backed up by Cloudinary's infrastructure

**Current adherence (audit, 08/05/2026):**
- РюЁ `/sinc` audio Рєњ Cloudinary, URL on `customer_communications.audio_url` AND `media_analyses.cloudinary_url`
- РюЁ `/intake` photos Рєњ Cloudinary, URL on `media_analyses.cloudinary_url` (NOT on `customer_communications.audio_url` Рђћ that column is audio-specific by name)
- РюЁ `/intake` videos Рєњ same as photos
- `[ ]` PDF intake (future) Рєњ must conform when built

- `[ ]` Marble sample uploads (future) Рєњ must conform when built
